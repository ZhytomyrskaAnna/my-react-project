**Тема:** Робота з URL-параметрами. Створення динамічних посилань та обробка фільтрів пошуку через Query String за допомогою React Router  
**Мета:** Ознайомитися з використанням хуків `useParams` та `useSearchParams` для синхронізації стану додатку з URL.  

## Створіть випадаючий список (<select>), який дозволятиме сортувати елементи "Від А до Я" або "Від Я до А" (або за датою).

``` jsx
    <select value={sortOrder} onChange={handleSortChange} style={{ padding: '8px' }}>
          <option value="asc">Від А до Я</option>
          <option value="desc">Від Я до А</option>
    </select>
```

## Зробіть так, щоб обране сортування зберігалося в URL (наприклад: /?query=react&sort=desc). Підказка: вам доведеться передавати існуючі параметри разом з новими при виклику setSearchParams.

``` jsx
    const handleSortChange = (e) => {
    const sort = e.target.value;
    const newParams = new URLSearchParams(searchParams);
    newParams.set("sort", sort);
    setSearchParams(newParams);
  };
```

## Контрольні запитання

**1. Поясніть принципову різницю між URL-параметрами шляху (Path Parameters, наприклад `/users/42`) та параметрами запиту (Query Parameters, наприклад `/users?sort=asc`). В яких випадках який підхід слід використовувати?**
Параметри шляху призначені для ідентифікації конкретного ресурсу, тому вони є невід’ємною частиною URL-адреси, без якої сторінка просто не відкриється. Наприклад, у маршруті /users/42 число 42 вказує на унікального користувача. Параметри запиту, що йдуть після знаку питання, є опціональними та служать для зміни представлення даних, таких як сортування чи фільтрація. Використовуйте параметри шляху для ієрархічних даних та обов'язкових об'єктів, а параметри запиту — для налаштувань вигляду списків, які користувач може змінювати за бажанням.
**2. Що робить хук `useParams()` і який обов'язковий синтаксис необхідно використати в визначенні `<Route path="...">` для того, щоб цей хук запрацював?**
Хук useParams дозволяє компоненту отримати доступ до динамічних частин URL-адреси, перетворюючи їх на зручний об'єкт із ключами та значеннями. Щоб цей механізм запрацював, у визначенні маршруту необхідно обов'язково використати двокрапку перед назвою змінної, наприклад path="/post/:postId". Саме цей символ повідомляє бібліотеці React Router, що частина шляху після слешу є динамічним параметром, а не статичним словом, і його потрібно зберегти в об'єкті параметрів.
**3. Чому зберігання стану пошукового рядка (Search Bar) в URL (через `useSearchParams`) вважається архітектурно кращим рішенням, ніж зберігання його у локальному стані компонента (`useState`)? Опишіть переваги для кінцевого користувача.**
Зберігання стану пошуку безпосередньо в URL-адресі перетворює сторінку на повноцінне посилання, яким можна поділитися з іншими людьми, зберігши всі застосовані фільтри. Це архітектурно краще за локальний стан useState, оскільки дозволяє користувачеві використовувати кнопки навігації браузера «Назад» і «Вперед» для перемикання між результатами пошуку. Такий підхід гарантує, що при оновленні сторінки або надсиланні посилання в месенджері стан інтерфейсу залишиться ідентичним, що є критично важливою вимогою для сучасних веб-додатків та інтернет-магазинів.
**4. Якщо ви використовуєте метод масиву `filter()` прямо перед поверненням JSX-розмітки, чи відбувається ця фільтрація при кожному рендері компонента? Як це може вплинути на продуктивність для дуже великих масивів (напр. 10 000+ елементів) і який інший React-хук ви б використали для оптимізації?**
Виклик методу filter() безпосередньо перед рендером призводить до повного перерахунку масиву при кожній зміні будь-якого стану в компоненті, що може серйозно уповільнити роботу інтерфейсу при обробці 10 000 і більше елементів. Для запобігання зайвим обчисленням та фрізам варто використовувати хук useMemo, який кешує відфільтрований результат і оновлює його лише тоді, коли змінюється вихідний масив або сам рядок пошуку. Це дозволяє уникнути повторного проходження по всьому масиву, якщо користувач просто перемикає інші елементи інтерфейсу, які не впливають на зміст списку.
**5. В яких об'єктах стандарту Web API у звичайному (базовому) JavaScript зберігаються параметри після знаку питання `?`, і як React Router спрощує роботу з ними?**
У стандартному JavaScript параметри після знаку питання доступні через об'єкт URLSearchParams, який створюється на основі рядка window.location.search. React Router значно спрощує цей процес, надаючи хук useSearchParams, який працює аналогічно до стандартного useState, але автоматично синхронізує дані з адресною стрічкою браузера. Це позбавляє розробника необхідності вручну парсити рядки, стежити за історією переміщень та оновлювати URL, оскільки бібліотека бере на себе всю низькорівневу логіку взаємодії з Web API.
